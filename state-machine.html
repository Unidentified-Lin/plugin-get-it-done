💥 崩潰復原（Step 2 — status=RUNNING AND batch_ended_at=null）

  • Sub-case 0：PLANNING singleton 崩潰 — <5min 視為仍在工作（[CRASH_WAIT] 退出等待）；≥5min 視為超時，重啟 PLANNING（[CRASH_DETECTED]）。
  • Sub-case A：claimed_set 非空 — sub-agents 已 spawn 但結果未持久化，用相同 id 重 spawn（Attempts 未增、scratch 以 task_id 為鍵、findings 乾淨覆寫，re-spawn 安全）。
  • Sub-case B：claimed_set 為空 — 結果已持久化但 batch envelope 未關閉，清理殘留 Claimed_by 並補關 batch。

🏁 Milestone 狀態派生（無持久化 Status 欄位，每 tick 重算）

  • validating — M.Claimed_by != null
  • pending — M.Tasks 內任一任務 Status != done
  • tasks_done — 全 done、無 validator in flight、且無 VR 或最近 VR 為 fail 未 escalate
  • validated — 最近 VR verdict == pass（解鎖下游 milestone）
  • blocked — 最近 VR escalate_to_blocked == true

⚙️ EXECUTING 批次優先序（greedy，單 tick，異質 ≤5）

  • P1 per-task validator — drain executed（最快解鎖下游 pending）
  • P2 milestone validator — tasks_done 的 milestone
  • P3 rework executor — needs_rework（最舊優先）
  • P4 new executor — deps 全 done 且 Milestone==active_ms 的 pending
  • 寫面互斥才同 batch；Touches 重疊則 [DEFER] 到下個 batch。

🟣 軟暫停 PLANNED_PAUSE（非 phase，是 soft exit reason）

  • milestone validator 對 PauseAfter:true 的 milestone 回 pass 時觸發。
  • 寫 [PLANNED_PAUSE] 到 progress_log 後乾淨 EXIT。
  • phase 維持 EXECUTING、status WAITING — 不需手動 unblock。
  • 使用者下次 /continue 自然接續下一個下游 milestone。
