# Task Queue

_Owned by the dispatcher for status / claim / attempts / validation-results writes. Planner writes the structure (task list, dependencies, milestones, PRD-Refs, descriptions). Executor and Validator do NOT edit this file in v2 — they emit their results via the `---agent-return---` YAML contract (see `team/state.md`)._

## Status Vocabulary

Per-task `Status` (single source of truth — supersedes the v1 `Validation Result` field):

```
pending         — not yet claimed; will be picked when all dependencies reach `done`
claimed         — dispatcher has spawned an executor for this task (set just before spawn)
executed        — executor returned `completed`; awaiting validator
validating      — dispatcher has spawned a validator for this task
done            — validator returned `verdict: pass`
needs_rework    — validator returned `verdict: fail` and did NOT escalate; will be re-spawned
blocked         — validator escalated `escalate_to_blocked: true`, or executor returned a blocker
                  → planner must intervene (phase falls back to PLANNING) or human steps in
```

Per-task fields filled / updated by the **dispatcher** (NOT by sub-agents directly):

```
Claimed_by   — slot identifier (e.g. exec-T-007 or val-T-007 or rework-T-007); null when not in flight
Claimed_at   — ISO timestamp when the dispatcher set Claimed_by; null otherwise
Attempts     — integer; incremented AFTER each executor return is persisted
Artifact     — path returned by executor (must live under team/workspace/exec-<task_id>/)
Validation Results — append-only list, one entry per validator return for this task
```

## Dependency Behavior (DAG)

Every task MUST have a `Dependencies:` field — a possibly-empty list of task IDs. A task is eligible to be claimed (transition `pending → claimed`) only when **every** listed dependency has `Status: done`. The dispatcher computes this each tick.

Planner self-checks the DAG before handing the task_queue to the dispatcher (no cycles, no self-refs, no references to non-existent task IDs). The dispatcher also runs a defensive DAG pre-check on every tick — if violations are found it falls phase back to `PLANNING` (so Planner can fix it) and writes a `[BAD_DAG]` line to `progress_log.md`. No batch is spawned until the DAG validates.

Tasks may also carry a `Milestone:` field (e.g. `M1`). When all tasks in a milestone reach `Status: done`, the dispatcher spawns one **milestone validator** to verify the milestone's overall acceptance criteria before any work in a downstream milestone is allowed to start. Milestone validators are not enabled in Stage 1 (single-track), but planner should still populate the field — it activates in Stage 3.

## Milestones (Stage 3+)

Planner writes one entry per milestone alongside the task list. The dispatcher computes milestone status from per-task `Status` + this section, and gates downstream-milestone tasks on prior-milestone validation.

```markdown
## Milestones

### M1: <milestone title>
- **Tasks**: [T-001, T-002, T-003]
- **Claimed_by**: null            # mval-M1 when a milestone validator is in flight; cleared by dispatcher on result
- **Claimed_at**: null
- **ValidatorAttempts**: 0
- **Acceptance Criteria**: free-form description of what "M1 done" means
                            at the integration / cross-task level (the
                            per-task metrics are individually verified by
                            per-task validators; this layer catches gaps
                            between tasks that no single task's criteria covers)
- **PauseAfter**: false           # optional; default false. When true AND a milestone validator passes this milestone,
                                  # the dispatcher emits [PLANNED_PAUSE] and EXITs cleanly (soft pause — phase stays
                                  # EXECUTING/WAITING so the next /continue resumes the downstream milestone). Use
                                  # ONLY at natural human-review boundaries (UX feel, real-world testing, external
                                  # sign-off) — see planner rule PR-019.
- **PauseReason**: null           # required when PauseAfter is true — one-line reason shown to the user at pause
- **Validation Results**: []     # appended each time a milestone validator returns
```

Milestone **status is not a persisted field** — it is **derived by the dispatcher on every tick** from (a) per-task statuses, (b) `Claimed_by`, (c) the latest `Validation Results` entry. The advantage is that status auto-resyncs when any task in the milestone changes — there is no sticky `failed` state that can lag behind a task going back to `needs_rework`.

### Derivation rule

```python
def milestone_status(M):
    if M.Claimed_by is not None:
        return "validating"
    if any(t.Status != "done" for t in M.Tasks):
        return "pending"
    # All tasks are done and no validator in flight.
    if not M.Validation_Results:
        return "tasks_done"
    latest = M.Validation_Results[-1]
    if latest.verdict == "pass":
        return "validated"
    if latest.escalate_to_blocked:
        return "blocked"
    # latest verdict was fail but tasks are all done again now (e.g. their
    # `task_ids_to_rework` flipped them to needs_rework, executors finished,
    # validators passed them, now back to done) → eligible for re-validation.
    return "tasks_done"
```

Status flow (induced, not persisted):

```
pending     → tasks_done   (every task in Tasks reaches Status: done)
tasks_done  → validating   (dispatcher claims it for a milestone validator)
validating  → validated    (milestone validator returns verdict: pass)
validating  → tasks_done   (validator returns fail with non-empty task_ids_to_rework — those tasks
                            flip to needs_rework, milestone derivation now sees them non-done so
                            actually flows back to `pending` until they reach done again, then
                            naturally returns to tasks_done for re-validation)
validating  → blocked      (validator returns fail with escalate_to_blocked: true; dispatcher
                            also flips phase to AWAITING_HUMAN)
*           → pending      (any task re-enters non-done at any time)
```

The "structural failure" case — validator returns `verdict: fail` with `task_ids_to_rework: []` and `escalate_to_blocked: false` — is handled by the dispatcher falling phase back to `PLANNING` (see Step 9). The persisted Validation Results entry remains; Planner reads it, decides whether to add/remove tasks or rewrite the milestone's Acceptance Criteria. Once Planner returns, the milestone's derived status is recomputed cleanly from the (possibly modified) task list.

### Milestone gate (downstream blocking)

A task in milestone `M_k` is eligible to be claimed by an executor (Step 5 P4 in `skills/continue/SKILL.md`) only when:

1. Its `Dependencies:` are all `Status: done`, AND
2. Every milestone `M_1..M_{k-1}` has derived `milestone_status() == "validated"`.

Milestone IDs are conventionally ordered alphanumerically (`M1 < M2 < M10`); dispatcher uses string-ascending comparison. Planner SHOULD use zero-padded IDs (`M01, M02, M10`) when the goal has ≥10 milestones to keep ordering monotonic.
